# Архитектура и Контекст для ИИ-агента (Antigravity IDE Integration)

Привет! Если ты читаешь это, значит мы продолжаем работу над проектом **Antigravity**. Это шпаргалка для тебя (ИИ-помощника), чтобы ты моментально вошёл в контекст сложной схемы маршрутизации между IDE, нашим кастомным Менеджером и серверами Google.

## 🏗 Как всё устроено (Текущая Архитектура)

Наша цель — заставить официальную IDE-интеграцию (Gemini Code Assist) работать через наш локальный Менеджер (Antigravity-Manager), написанный на Rust/Axum + Tauri. 

**Цепочка запросов выглядит так:**
`IDE (на Windows)  -->  Node.js Прокси (в WSL, порт 8046)  -->  Antigravity-Manager (в WSL, порт 8055)  -->  Google Backend`

### Почему именно такая цепочка? (Исторический контекст)

1. **Конфликт портов (Windows vs WSL):**
   Исторически Менеджер должен был работать на порту `8045`. Однако, в Windows-среде "залип" процесс `antigravity_tools.exe`, который захватил порт `8045` на `127.0.0.1`. Из-за специфики сети WSL, мы не могли "отвоевать" этот порт изнутри Ubuntu.
   * **Решение:** В исходном коде Менеджера (`src-tauri/src/proxy/server.rs`) мы добавили принудительное переопределение порта через переменную среды `ABV_PORT`. Теперь Менеджер запускается на порту `8055` (`export ABV_PORT=8055`).

2. **Проблема маршрутизации Axum (`/v1internal:loadCodeAssist`):**
   IDE делает запросы с двоеточием в URL (например, `/v1internal:loadCodeAssist`). Фреймворк **Axum**, на котором написан Менеджер, интерпретирует двоеточие как *динамический параметр пути* (переменную). Это приводило к тому, что роутер сходил с ума и возвращал 404.
   * **Решение:** Появился **Node.js Прокси (8046)** (`scripts/proxy8046.js`). Его задача — принять запрос от IDE, **заменить двоеточие на слеш** (получается `/v1internal/loadCodeAssist`) и отправить на порт 8055.
   * Менеджер (в `server.rs` и `proxy/handlers/...`) настроен на приём маршрута `/v1internal/*path`. Когда запрос попадает в fallback-обработчик Менеджера, он делает обратную замену (`/v1internal/` -> `/v1internal:`) перед пересылкой оригинального запроса на сервера Google (`cloudcode-pa.googleapis.com`).

3. **Проблема 404 при авторизации (`userinfo` и `tokeninfo`):**
   Прежде чем IDE начнёт слать запросы на генерацию кода, она проверяет валидность токена, обращаясь к эндпоинтам OAuth (например, `/oauth2/v2/userinfo`). Менеджер эти роуты не обрабатывает, так как они предназначены для `www.googleapis.com`, а не для Cloud Code API.
   * **Решение:** Node.js Прокси перехватывает запросы, содержащие `tokeninfo` или `userinfo`. Вместо того чтобы слать их в Менеджер, прокси **локально отвечает `200 OK`** и возвращает фейковый, но валидный JSON с данными профиля пользователя. IDE считает, что авторизация пройдена успешно.

## 🛠 Где лежит код и как его запускать

- **Antigravity-Manager:** `/home/yaaaa/projects/AG-manager/Antigravity-Manager/`
  - Роутинг и переопределение портов: `src-tauri/src/proxy/server.rs`
  - Запуск: 
    ```bash
    cd /home/yaaaa/projects/AG-manager/Antigravity-Manager
    export ABV_PORT=8055
    xvfb-run -a npm run tauri dev
    ```

- **Node.js Прокси:** В репозитории Менеджера сохранен как `scripts/proxy8046.js` (скопирован из корня).
  - Запуск:
    ```bash
    node /home/yaaaa/projects/proxy8046.js
    ```
  - В базе данных SQLite клиентской IDE прокси-адрес (`code_assist_proxy_url`) должен быть жестко прописан на `http://127.0.0.1:8046`.
